Philosophy

AI-Native From the Ground Up

  • Not bolted on. AI isn't a feature added to a scheduling tool. It's the architecture.
  • Clojure-native. The LLM pipeline (DSCloj), the event store (Grain), and the behavior trees (ORC) are all Clojure. No Python glue layer for day-to-day work.
  • Event-sourced truth. Everything that happens is an immutable fact. Vivian reasons over events, not mutable state.
  • Polylith everywhere. One architecture pattern across all services — composable components that snap together.
  • Self-improving. ORC behavior trees are trainable. GEPA closes the optimization loop.

Grain · Multi-Tenancy + Cache

Built for 500 Clinics

  • :tenant-id on every operation. Postgres Row-Level Security + per-tenant advisory locks. Two tenants never share a cache entry.
  • L1 cache — in-process LRU atom. Zero serialization. Sub-millisecond reads. TTL configurable per read model.
  • L2 cache — LMDB on disk, Fressian-serialized, survives restarts. Watermark-based — only replays events since last checkpoint.
  • 3 L2 strategies auto-selected: Monolithic (<10K keys) · Segmented (≥10K, 64 hash segments) · Partitioned (per entity, e.g. patients-per-clinic)

Multi-tenancy isn't a feature. It's the invariant. Every layer enforces it.

Grain · AI/ML Extensions

AI is a First-Class Citizen

  • poly/behavior-tree-v2 — behavior tree engine component baked into Grain. ORC's sheet-service runs on top of this.
  • grain-dspy-extensions — libpython-clj2 bridge to Python DSPy. defsignature macro auto-generates Pydantic + DSPy Signature classes from Malli schemas. Escape hatch for genuine Python ML (GEPA optimization).
  • DSCloj integration — primary path for LLM calls. Malli schemas unify the type system across commands, queries, read models, and LLM I/O.
  • Langfuse — external observability component. Token tracking, trace storage, evaluation integration.
sheet-service (ORC) · Grain Consumer Clojure Production

The Behavior Tree Engine

ORC = Orchestral. sheet-service is a Grain service component — one peer among XiVara's 17, not a separate layer.

  • Node types: leaf, sequence, fallback, condition, LLM-condition, parallel, map-each
  • Executors: AI (LLM via DSCloj), code (Clojure), tool (MCP)
  • Blackboard: keyed data context with Malli schemas — shared state across a tree run
  • Versioning: draft/published modes, stashes, full execution tracing with per-node timing
  • Retry/timeout/repeat decorators — production-grade reliability on every node

ORC Vision — Self-Improving Behaviors

Skills → Trees → Optimization Loop

  • Skills → Trees Pipeline — Skills become executable behavior trees. MCP tools researched via Tavily/Exa. Skill steps map to ORC nodes.
  • Rolling Metric Monitoring — Event store tracks rolling averages; threshold breaches trigger failure analysis + auto-training.
  • Tree Self-Description — Trees describe what they can solve. Builds a searchable library/ontology of capabilities.
  • Best-of-N Generation — Generate N candidate trees, score against objectives, refine iteratively.
  • Conversational Debugging — "Talk to traces" — load execution context, ask questions about failures.
  • Personality Layer — Message filtering for Vivian's voice/tone.

Product Insight

Two Vivians

Product Vivian
Lives inside the clinic. Knows appointments, patients, checkout, clinical notes. Knows nothing about XiVara the company. XiVara's intent encoded at infrastructure level — not runtime accessible.
Elevated Vivian (concept)
Built on top of product Vivian. Serves the XiVara team. Knows the product, the architecture, the gaps. Translates between technical and product layers. Carl is the current prototype for this.

The epistemic boundary that defines product Vivian creates the need for the elevated layer by definition.

The Product

How This Becomes Vivian

  • Grain's event store = every patient interaction, booking, payment, inventory change — immutable facts Vivian reasons over
  • Grain's projections = read models like "patients at churn risk," "Tuesday capacity," "Juvederm inventory" — always current, sub-millisecond
  • DSCloj modules = Vivian's intelligence units — typed LLM calls with validated outputs. "Given these events, what's the recommended action?"
  • ORC trees = Vivian's workflows — the actual logic of what she does. Trainable, versioned, self-describing.
  • GEPA = how Vivian gets smarter — her behavior trees tune themselves on real clinic outcomes
